# LeadCMS SDK v3.6.0 - Redirects, Content Creation, and Status Clarity

This release adds full redirect management (pull, push, status, and nginx map generation), an interactive `add-content` wizard, a `--delete` flag for push commands, comment reparenting support, and a complete overhaul of status display labels to make sync direction unambiguous.

## What's New

### Redirects Management (New Feature)

Redirects are now a fully managed entity type in the SDK. Four new CLI commands cover the complete lifecycle:

#### `leadcms pull-redirects`

Redirects are stored in **language-scoped YAML files** that mirror the content folder structure:

- `redirects/redirects.yaml` — default-language and path-based (language-agnostic) redirects
- `redirects/{lang}/redirects.yaml` — redirects whose source is a non-default language (e.g. `redirects/de/redirects.yaml`)

Within each file, `fromLanguage` is **omitted** — it is implied by the folder. `toLanguage` is also omitted when it equals the folder's language, but **kept** when the target is a different language (cross-language redirect).

The YAML groups entries by kind (`permanent` for 301, `temporary` for 302). Each item uses named fields for source and target — `sourceType` and `targetType` are **never written to disk**; they are auto-detected from which fields are populated when you push.

```yaml
# redirects/redirects.yaml  (default language "en" + path-based)
permanent: # 301 redirects
  - fromPath: /old-page # InternalPath source (no language)
    toPath: /new-page

  - fromSlug: old-article # ContentSlug — fromLanguage "en" implied by folder
    toSlug: new-article # toLanguage "en" implied, omitted

  - fromSlug: old-article # Cross-language redirect: fromLanguage "en" implied
    toSlug: artikel # toLanguage kept because it differs from folder lang
    toLanguage: de

  - fromContentId: 42 # ContentId source
    toPath: /new-page

temporary: # 302 redirects
  - fromPath: /promo
    toUrl: https://example.com/sale # ExternalUrl target
```

```yaml
# redirects/de/redirects.yaml  (German redirects)
permanent:
  - fromSlug: alter-artikel # fromLanguage "de" implied by folder
    toSlug: neuer-artikel # toLanguage "de" implied, omitted

  - fromSlug: alter-artikel # Cross-language: fromLanguage "de" implied
    toSlug: old-article # toLanguage "en" kept — differs from folder
    toLanguage: en
```

**Source fields** (exactly one set): `fromPath` → `InternalPath`; `fromSlug` (+folder-implied `fromLanguage`) → `ContentSlug`; `fromContentId` → `ContentId`

**Target fields** (exactly one set): `toUrl` → `ExternalUrl`; `toPath` → `InternalPath`; `toSlug` (+optional `toLanguage`) → `ContentSlug`; `toContentId` → `ContentId`

Options: `--reset` (delete local files and sync token, then pull from scratch), `-r <remote>` (multi-remote target).

#### `leadcms push-redirects`

Reads all language-scoped redirect files from `redirects/` (root file + any `{lang}/redirects.yaml` subdirs), compares against remote state using surrogate keys, and creates or updates redirect entries. After a successful push, the local files are updated with server-assigned `id` and `updatedAt` values.

Options: `--force`, `--dry-run`, `--delete` (remove remote redirects not present locally), `-r <remote>`.

#### `leadcms status-redirects`

Shows redirect sync status (added locally / updated locally / added remotely / conflict) without making changes. Redirects that exist only on the remote are always shown without needing `--delete`.

Options: `--delete` (also surface deletion candidates), `-r <remote>`.

#### `leadcms generate-redirects-map`

Generates nginx `map` files from all language-scoped redirect files — no remote connection required. Outputs `301.map` and `302.map` for use in nginx configuration. `ContentSlug` and `ContentId` sources are resolved to paths using the `redirects.pathPattern` config option.

**Language mode is inferred from `pathPattern`:**

- **`{language}` in `pathPattern`** (e.g. `"/{language}/{slug}"`) — multi-language-per-domain setup. All language folders are read and **every redirect appears in the map** regardless of language. A single nginx server handles all languages via path-prefixed URLs.
- **No `{language}` in `pathPattern`** (e.g. `"/{slug}"`) — single-language-per-domain setup. Only redirects matching the active language are included (from `--language` flag, `LEADCMS_DEFAULT_LANGUAGE` env var, or `config.defaultLanguage`).

The `--language` flag can always be used to narrow the output explicitly, even when `pathPattern` contains `{language}`.

Options: `--output, -o <file>`, `--language, -l <lang>`, `--dry-run, -d`.

### Interactive Content Creation: `leadcms add-content`

A new interactive wizard for creating content files locally:

```bash
npx leadcms add-content [slug]
```

The wizard:

1. Accepts an optional positional slug (or prompts for one)
2. Fetches available content types from the remote and lets you choose
3. Prompts for required frontmatter fields (title, description, author, language, category, tags)
4. Writes an MDX or JSON file to the correct directory

The resulting file is immediately ready to push with `npx leadcms push`.

### `--delete` Flag for Push Commands

Content push now supports explicit deletion of remote items that no longer exist locally:

```bash
npx leadcms push --delete
```

Without `--delete`, items that exist only on the remote are shown as `added remotely:` so developers can pull them intentionally. With `--delete`, local files are treated as the source of truth: remote items missing locally are shown as `deleted locally:` and can be removed from LeadCMS.

This keeps the recovery path straightforward even if a local metadata file was cleaned up along with a deleted content file.

The flag is also available on all focused push commands: `push-email-templates`, `push-segments`, `push-sequences`, `push-redirects`, `push-settings`.

### Comment Reparenting

Comments now track and push `parentId` and `commentableId`. When a comment's parent or target entity changes between pulls, the SDK detects the difference and includes both fields in the update payload. Conflict detection also compares these fields, so reparented comments are correctly flagged rather than silently overwritten.

---

### Remote-Only Entities Always Visible in Status

Previously, running `leadcms status` (or any entity-specific status command) without `--delete` would silently hide records that existed on the remote but had no local counterpart. This made it easy to miss content, media, email templates, segments, sequences, comments, and redirects that had been created on the remote and needed to be pulled.

All entity types now surface remote-only records in normal status output as `added remotely:`, regardless of whether `--delete` is passed:

- **Content** — new `remoteCreated` bucket in `ContentOperations`; status no longer short-circuits when the local content directory is empty
- **Media** — continues checking the remote even when the local media directory is empty; reports `added remotely:` for files present only on the remote
- **Email templates, segments, sequences** — remote-only records are detected after the main matching pass and reported separately
- **Comments** — remote-only comments are detected using the same matched-ID tracking used for conflict detection, avoiding false positives with translation-key-matched threads
- **Redirects** — remote-only redirects were already fixed; no additional changes required this release

### Explicit Directional Display Labels

All status and push output labels have been renamed to make the direction of each change unambiguous:

| Old label           | New label                     |
| ------------------- | ----------------------------- |
| `new:`              | `added locally:`              |
| `modified:`         | `updated locally:`            |
| `deleted:`          | `deleted locally:`            |
| `renamed:`          | `renamed locally:`            |
| `type change:`      | `type changed locally:`       |
| `new remotely:`     | `added remotely:`             |
| `updated remotely:` | _(unchanged)_                 |
| `deleted remotely:` | _(unchanged)_                 |
| `conflict:`         | _(unchanged — bidirectional)_ |

### Redirect Output: Language and Slug Shown Separately

Slug-based redirects are now displayed with the language code as a distinct bracket token, consistent with how content is rendered in `status-all` and `push-all` output.

Before:

```
added remotely: [302] [ru/company-history] → [ru-RU/company-history]
```

After:

```
added remotely: [302] [ru] company-history → [ru-RU] company-history
```

Raw path and URL redirects (`fromPath`, `toPath`, `toUrl`) are not affected.

## Bug Fixes

### Fixed: Remote-only records counted toward push confirmation total

In joined `leadcms push`, remote-only entries (`added remotely:`) were included when calculating the number of pending changes shown in the confirmation prompt ("Proceed with pushing N changes?"), which made the count misleading because those records require a pull, not a push. Remote-only entries are now surfaced in the push summary for visibility but are excluded from the actionable push count that drives the confirmation prompt. A new `push-all-counts.ts` utility centralises this logic and is covered by dedicated tests.

### Fixed: Settings `deleted locally:` shown in green instead of red

The settings aggregate renderer used a success (green) colour for `deleted locally:` entries. This now uses the same red conflict colour used by other entity types for deletion candidates.

### Fixed: Status commands reporting "all in sync" when remote had records and local was empty

When the local redirects file was absent or the local content directory was empty, status commands would skip the remote comparison entirely and report no changes. The root cause was an early return before the remote-only detection pass. All affected entity types have been corrected.

### Fixed: Push refreshing unrelated local files after updates

Push commands now refresh just-touched local files from create/update API responses instead of running broad post-push pulls. This keeps server-controlled values such as `id`, `updatedAt`, and normalized body/metadata aligned for content, comments, email templates, segments, sequences, and media without importing unrelated remote changes during `push` or joined `push-all` runs.

### Fixed: Unparseable local content being treated as missing

Content push/status now stops with a clear local-file error when an MDX or JSON file cannot be parsed, including unresolved Git conflict markers in frontmatter. Unified `leadcms push` also surfaces this content blocker instead of omitting the Content section. Broken local files are no longer silently skipped, which prevents already-local content from being misreported as `added remotely` or disappearing from the report.

### Fixed: Media sync tests running at full network speed (~12s) in CI

The `fetchDeletedMedia` lookup inside `statusMedia` was not injectable, causing tests to hit real (or slow mock) sync endpoints. A `fetchDeletedMedia` dependency was added to `MediaDependencies` with the real implementation as the default, allowing test suites to inject `async () => []` and drop per-suite runtime from ~12s to ~0.3s.
