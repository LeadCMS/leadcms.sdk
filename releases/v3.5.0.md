# LeadCMS SDK v3.5.0 - Multi-Remote State Finalization

Named remotes are now supported consistently across login, pull, push, status, and reset workflows, with isolated sync state per remote and stable per-remote ID and metadata storage.

## What's New

### Named Remotes Across CLI Workflows

Most sync and auth commands now accept `--remote` / `-r` and resolve the selected CMS instance through the configured `remotes` block and `defaultRemote`.

Examples:

```bash
leadcms login -r dev
leadcms pull -r prod
leadcms status-content -r dev
leadcms push-email-templates -r prod
```

The `leadcms remote` command also supports managing and inspecting remotes:

- `list`
- `add`
- `remove`
- `show`
- `set-default`
- `reset`

### Per-Remote Sync State

Sync tokens are now stored per remote under `.leadcms/remotes/<name>/`, so pulling from one instance no longer overwrites another instance's incremental sync state.

Current state files:

```text
.leadcms/
  remotes/
    prod/
      content-sync-token
      media-sync-token
      comments-sync-token
      email-templates-sync-token
      metadata.json
```

### Per-Remote Metadata Map

The SDK now stores remote-specific IDs and timestamps for both content and email templates in a single per-remote file.

Final `metadata.json` structure:

```json
{
  "content": {
    "en": {
      "docs/sdk-api/cli-basics": {
        "id": 77,
        "createdAt": "2026-01-15T10:00:00Z",
        "updatedAt": "2026-03-01T14:30:00Z"
      },
      "docs/sdk-api/pull-content": {
        "id": 78
      }
    }
  },
  "emailTemplates": {
    "en": {
      "WelcomeEmail": {
        "id": 42,
        "updatedAt": "2026-03-02T09:15:00Z"
      },
      "WeeklyDigest": {
        "id": 43
      }
    }
  }
}
```

### Stable Ordering In Generated State Files

Language blocks and slug/template-name entries are written in alphabetical order. This keeps `metadata.json` stable across regenerations and reduces noisy diffs.

### Default-Remote Metadata Preserved In Local Files

When pulling from a non-default remote:

- content frontmatter `id`, `createdAt`, and `updatedAt` continue to reflect the default remote
- email template frontmatter/comment metadata `id`, `createdAt`, and `updatedAt` also continue to reflect the default remote
- the active remote's own IDs and timestamps are stored only in that remote's `metadata.json`

This keeps local files framework-friendly and stable while still allowing correct per-remote push matching and conflict detection.

### Remote-Aware Push And Status Baselines

Push and status workflows now use the selected remote's own ID and metadata maps instead of relying on whichever IDs and timestamps happen to be present in local files. This prevents cross-remote mismatches and false conflicts when switching between environments.

## Bug Fixes

### Fixed: Non-default remote pulls dropping metadata in email template files

Pulling email templates from a non-default remote could fail to restore the default remote's `id`, `createdAt`, and `updatedAt` into local template files after the map refactor. The lookup now uses the final nested map structure correctly.

## Test Coverage

- 957 tests passing in the full suite
- Integration coverage for default-remote metadata behavior in both content and email template pulls

## Upgrade Notes

If you tested an unreleased pre-3.5 multi-remote build that wrote the older flat map shape, regenerate remote state using one of the following:

```bash
leadcms remote reset <name>
# or
leadcms pull --reset -r <name>
```

This release does not preserve the unreleased pre-3.5 state file format.
